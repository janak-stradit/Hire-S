HIREX DATABASE - CANDIDATE DATA TABLES AND COLUMNS
===================================================

Database: HireX
Database engine: PostgreSQL
Schema: public
Verified against the live PostgreSQL database: 2026-06-19

PURPOSE
-------
This document lists the database tables and columns used to store candidate
accounts, profiles, resumes, jobs, applications, parsed resume/job data, and
pre-screen validator results.


1. TABLE: users
===============
Purpose: Stores authentication and account-level information for candidates,
recruiters, HR users, and administrators.

Columns:
1. id                         - character varying - NOT NULL - Primary key
2. email                      - character varying - NOT NULL - Unique login email
3. password_hash              - character varying - NOT NULL - Hashed password
4. role                       - character varying - NOT NULL - candidate/recruiter/hr/admin
5. is_active                  - boolean           - NOT NULL - Account active status
6. created_at                 - timestamp with time zone - NOT NULL - Account creation time

Candidate relationship:
users.id -> candidate_profiles.candidate_id


2. TABLE: candidate_profiles
=============================
Purpose: Stores the candidate's personal, employment, education, salary, and
professional-link information.

Columns:
1. candidate_id               - character varying - NOT NULL - Primary key; FK to users.id
2. first_name                 - character varying - NULL
3. last_name                  - character varying - NULL
4. phone                      - character varying - NULL
5. city                       - character varying - NULL
6. state                      - character varying - NULL
7. country                    - character varying - NULL
8. current_company            - character varying - NULL
9. current_role               - character varying - NULL
10. total_experience          - double precision  - NULL - Candidate experience in years
11. expected_salary           - integer           - NULL
12. notice_period             - character varying - NULL
13. highest_education         - character varying - NULL
14. linkedin_url              - character varying - NULL
15. github_url                - character varying - NULL
16. portfolio_url             - character varying - NULL
17. profile_completion_percentage - integer       - NOT NULL


3. TABLE: resumes
================
Purpose: Stores uploaded resume file metadata. Resume parsing output is stored
separately in parsed_resumes.

Columns:
1. resume_id                  - character varying - NOT NULL - Primary key
2. candidate_id               - character varying - NOT NULL - FK to candidate_profiles
3. filename                   - character varying - NOT NULL - Stored file name
4. original_filename          - character varying - NOT NULL - Candidate's original file name
5. file_size                  - integer           - NOT NULL - Size in bytes
6. file_type                  - character varying - NOT NULL - PDF/DOCX MIME type
7. storage_path               - character varying - NOT NULL - File storage location
8. uploaded_at                - timestamp with time zone - NOT NULL


4. TABLE: jobs
=============
Purpose: Stores job openings that candidates can browse and apply for.

Columns:
1. job_id                     - character varying - NOT NULL - Primary key
2. title                      - character varying - NOT NULL
3. department                 - character varying - NOT NULL
4. location                   - character varying - NOT NULL
5. employment_type            - character varying - NOT NULL
6. experience_min             - integer           - NULL
7. experience_max             - integer           - NULL
8. salary_min                 - integer           - NULL
9. salary_max                 - integer           - NULL
10. description               - text              - NOT NULL
11. skills_required           - text              - NOT NULL
12. status                    - character varying - NOT NULL - Example: open/closed
13. created_by                - character varying - NOT NULL - FK to users.id


5. TABLE: applications
======================
Purpose: Connects a candidate to a job and tracks the application state.

Columns:
1. application_id             - character varying - NOT NULL - Primary key
2. candidate_id               - character varying - NOT NULL - FK to candidate_profiles
3. job_id                     - character varying - NOT NULL - FK to jobs
4. application_status         - character varying - NOT NULL
5. applied_at                 - timestamp with time zone - NOT NULL

Application status values supported by the Phase 1 design:
- Applied
- Under Review
- Shortlisted
- Rejected
- R1 Scheduled
- R1 Completed
- T1 Scheduled
- T1 Completed
- Final Review

Constraint:
A candidate can apply to the same job only once.


6. TABLE: parsed_resumes
========================
Purpose: Stores structured information extracted from a candidate resume by
the Phase 2 pre-screen validator.

Columns:
1. parsed_resume_id           - character varying - NOT NULL - Primary key
2. resume_id                  - character varying - NOT NULL - FK to resumes
3. candidate_id               - character varying - NOT NULL - FK to candidate_profiles
4. skills                     - text              - NOT NULL
5. total_experience_years     - double precision  - NOT NULL
6. education                  - text              - NOT NULL
7. certifications             - text              - NOT NULL
8. projects                   - text              - NOT NULL
9. raw_text                   - text              - NOT NULL - Extracted resume text
10. parsed_at                 - timestamp with time zone - NOT NULL


7. TABLE: parsed_job_descriptions
==================================
Purpose: Stores structured requirements extracted from a job description.

Columns:
1. parsed_jd_id               - character varying - NOT NULL - Primary key
2. job_id                     - character varying - NOT NULL - FK to jobs
3. required_skills            - text              - NOT NULL
4. preferred_skills           - text              - NOT NULL
5. experience_min             - integer           - NULL
6. experience_max             - integer           - NULL
7. education_requirements     - text              - NOT NULL
8. certifications             - text              - NOT NULL
9. raw_text                   - text              - NOT NULL
10. parsed_at                 - timestamp with time zone - NOT NULL


8. TABLE: validator_results
============================
Purpose: Stores the candidate-to-job pre-screen evaluation, score breakdown,
decision, queue destination, and explanation.

Columns:
1. validator_result_id        - character varying - NOT NULL - Primary key
2. application_id             - character varying - NOT NULL - FK to applications
3. candidate_id               - character varying - NOT NULL - FK to candidate_profiles
4. job_id                     - character varying - NOT NULL - FK to jobs
5. parsed_resume_id           - character varying - NOT NULL - FK to parsed_resumes
6. parsed_jd_id               - character varying - NOT NULL - FK to parsed_job_descriptions
7. skill_score                - double precision  - NOT NULL - 0 to 100
8. experience_score           - double precision  - NOT NULL - 0 to 100
9. education_score            - double precision  - NOT NULL - 0 to 100
10. certification_score       - double precision  - NOT NULL - 0 to 100
11. keyword_score             - double precision  - NOT NULL - 0 to 100
12. final_score               - double precision  - NOT NULL - Weighted score, 0 to 100
13. decision                  - character varying - NOT NULL - PASS/REVIEW/FAIL
14. queue_target              - character varying - NOT NULL
15. explanation               - text              - NOT NULL
16. evaluated_at              - timestamp with time zone - NOT NULL

Decision and queue mapping:
- PASS   -> R1 Queue
- REVIEW -> HR Review Queue
- FAIL   -> Auto Reject Queue


CANDIDATE DATA FLOW ACROSS TABLES
=================================

users.id
  -> candidate_profiles.candidate_id
  -> resumes.candidate_id
  -> parsed_resumes.candidate_id
  -> applications.candidate_id
  -> validator_results.candidate_id

jobs.job_id
  -> applications.job_id
  -> parsed_job_descriptions.job_id
  -> validator_results.job_id

applications.application_id
  -> validator_results.application_id

resumes.resume_id
  -> parsed_resumes.resume_id


SUMMARY
=======

Candidate identity and login:          users
Candidate personal/profile details:    candidate_profiles
Candidate resume file metadata:        resumes
Candidate parsed resume information:   parsed_resumes
Available job information:             jobs
Parsed job requirements:               parsed_job_descriptions
Candidate job applications:            applications
Candidate pre-screen scores/results:   validator_results
